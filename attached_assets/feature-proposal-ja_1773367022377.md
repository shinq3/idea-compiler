# 機能提案

## 機能一覧（Must / Should / Could）

### Must
- **予約管理機能**
  - 目的: WebおよびLINEでイベント一覧を表示し、参加者の予約をスムーズに行う。［出典: プロジェクト概要 – In-Scope］
  - 対象ユーザー: イベント参加者（Web/LINE）
  - 根拠: RFP要件定義; WeCALLベンチマーク（https://generosity.co.jp/tech-design/wecall）

- **チェックイン機能**
  - 目的: QRスキャンまたは手動入力による当日来場者チェックインを実現し、正確な参加者管理を行う。［出典: プロジェクト概要 – In-Scope］
  - 対象ユーザー: スタッフ、参加者
  - 根拠: WeCALL「予約/チェックイン」モジュール

- **整理券発券機能**
  - 目的: 順番受付整理券をリアルタイムで発券・呼び出し通知し、待ち時間と混雑を最小化する。［出典: プロジェクト概要 – In-Scope］
  - 対象ユーザー: 参加者、スタッフ
  - 根拠: WeCALL「整理券機能」

- **オンライン決済連携**
  - 目的: クレジットカードやQRウォレットによるスマホ完結型決済を提供する。［出典: プロジェクト概要 – Objectives］
  - 対象ユーザー: 決済を行う参加者
  - 根拠: 顧客目的#3; RFPドラフト

### Should
- **リアルタイム同期**
  - 目的: 注文状況、整理券呼び出し、チェックイン情報をスタッフ端末と管理画面にリアルタイム反映する。［出典: プロジェクト概要 – Overview］
  - 対象ユーザー: スタッフ、管理者
  - 根拠: RFP; キックオフ会議ノート

- **管理者ダッシュボード**
  - 目的: 売上レポート、参加者分析、運用データの可視化を提供し、運営効率を向上させる。［出典: プロジェクト概要 – Objectives］
  - 対象ユーザー: イベント運営管理者
  - 根拠: 機能候補リスト; Objective#4

- **マルチチャネルUI**
  - 目的: LINEおよびWebで統一された利用体験を提供し、参加者のアクセス手段を多様化する。［出典: プロジェクト概要 – Scope］
  - 対象ユーザー: 参加者
  - 根拠: In-Scope項目

### Could
- **モバイルオーダー機能**
  - 目的: 参加者が物販を事前注文できるようにし、在庫管理とオペレーションを最適化する。［出典: 要件 – nice_to_have］
  - 対象ユーザー: 参加者、スタッフ
  - 根拠: WeCALLモバイルオーダー機能; RFP

- **プッシュ通知サービス**
  - 目的: リマインダーやプロモーション通知を配信し、参加率向上とキャンセル削減を図る。［出典: 機能候補リスト］
  - 対象ユーザー: 参加者
  - 根拠: 機能候補リスト

- **データ可視化・分析機能**
  - 目的: 予約・来場・注文データをグラフィカルに分析し、運営改善に活用する。［出典: 機能候補リスト］
  - 対象ユーザー: 管理者
  - 根拠: 機能候補リスト

- **柔軟な構成・モジュール独立性**
  - 目的: イベントごとの機能選択やモジュール単位での有効化/無効化を可能にする。［出典: 機能候補リスト］
  - 対象ユーザー: 管理者
  - 根拠: 機能候補リスト

## 受け入れ基準
1. Must機能がすべて実装され、受け入れテストに合格すること。
2. 予約、チェックイン、整理券発券ワークフローが負荷テストで安定稼働すること。
3. サンドボックス環境で複数の決済プロバイダによる決済が成功すること。
4. リアルタイム更新が2秒以内にすべてのクライアントに反映されること。
5. 管理者ダッシュボードに定義されたKPIが正確に表示されること。
6. UIコンポーネントがアクセシビリティとユーザビリティ基準を満たすこと。
